# Feature Specification: 飞书 AI 机器人 (Feishu Agent Bot)

**Feature Branch**: `001-feishu-agent-bot`

**Status**: Draft

---

## Overview

在飞书工作空间内提供一个可通过 @ 唤起的 AI 助手机器人：支持群聊和私聊，按用户独立维护对话上下文，实时展示"思考过程/正文/工具调用"三类信息，能调用网络搜索、网页抓取、飞书文档读取等工具以及外部 MCP 工具服务。

本次迭代**不包含 Skill（技能）系统**。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 群内 @ 机器人获得流式 AI 回答 (Priority: P1)

团队成员在业务群里通过 `@机器人 <问题>` 发起提问，希望看到机器人"边想边答"的过程，包括推理思考、正文回答，以及必要时查询工具的动作，整个过程在同一条消息卡片内实时更新。

**Why this priority**: 这是机器人最核心的使用场景，没有它其他所有功能都没有承载。团队成员在群里可直接对话才构成"群内 AI 助手"的价值主张。

**Independent Test**: 在测试群中添加机器人，发送 `@bot 介绍一下你自己`，观察：收到一条消息卡片、卡片内容在 1 秒内开始流式更新、回答完整后卡片停止更新且不再变化。

**Acceptance Scenarios**:

1. **Given** 机器人已加入群聊且用户拥有对话权限，**When** 用户发送 `@机器人 今天有什么新闻`，**Then** 系统在 3 秒内回复一张消息卡片，卡片先显示"思考中"占位，随后流式追加正文，若涉及联网查询则显示工具调用条目及其状态（进行中/成功/失败）。
2. **Given** 机器人正在回答用户 A 的问题，**When** 用户 B 在同一群里 @ 机器人发问，**Then** 用户 B 的对话独立处理，不与用户 A 的上下文或输出混杂。
3. **Given** 用户在群里未 @ 机器人，**When** 用户发送普通消息，**Then** 机器人不回应、不计费、不写入任何上下文。
4. **Given** 机器人无法正常回答（下游服务超时或报错），**When** 错误发生，**Then** 卡片内给出对用户可理解的失败提示（而非静默消失或暴露栈信息）。

### User Story 2 - 私聊对话 (Priority: P1)

成员直接打开与机器人的私聊对话框提问，获得与群聊一致的体验，但无需 @。

**Why this priority**: 私聊是对 1-1 私密问询、敏感资料查询的主要入口，体验与群聊等价且更常用。与 Story 1 是平行关系，单独上线任一即可提供价值。

**Independent Test**: 与机器人建立 1-1 会话，发送任意问题，观察是否返回流式卡片回复；发送第二条问题，确认机器人引用了上一条的上下文。

**Acceptance Scenarios**:

1. **Given** 用户与机器人处于 1-1 会话，**When** 用户发送任何文本消息，**Then** 机器人按 Story 1 相同的流式卡片格式回复，无需 @ 前缀。
2. **Given** 用户在私聊中连续发送两条相关问题，**When** 第二条发出，**Then** 机器人的回答体现对第一条内容的记忆（同一上下文）。

---

### User Story 3 - 会话控制与切换 (Priority: P2)

用户通过固定指令管理自己的会话：清空历史、切换模型、查看可用工具等。这些指令**不应**消耗 AI 能力配额或触发任何推理。

**Why this priority**: 长时间会话后用户需要"清空重开"或调试。没有 P1 的基础对话能力，本故事无法单独交付价值，因此排 P2。

**Independent Test**: 在已建立会话的情况下发送 `/clear`，再发问，验证机器人不会引用 `/clear` 之前的任何内容。

**Acceptance Scenarios**:

1. **Given** 用户与机器人已有若干轮对话，**When** 用户发送 `/clear`，**Then** 机器人立即（< 1 秒）回复确认"已清空上下文"，且后续消息不再携带清空前的记忆。
2. **Given** 用户在多模型环境下使用，**When** 用户发送 `/model <模型别名>`，**Then** 机器人确认切换成功；下一条问题使用新模型回答；若模型别名不存在则返回可用列表。
3. **Given** 用户想了解机器人能力，**When** 用户发送 `/help`，**Then** 机器人返回当前可用指令与可用工具摘要；发送 `/tools` 则返回详细工具列表及其当前连通状态。
4. **Given** 用户发送任何 `/` 开头的指令，**When** 指令被识别，**Then** 不触发 LLM 调用、不计入上下文、不占用消息速率配额。

### User Story 4 - 工具辅助的信息查询 (Priority: P2)

用户的问题需要机器人访问外部信息（最新资讯、特定网页内容、飞书云文档内容）时，机器人应自动调用相应工具并把结果整合进回答，过程对用户可见。

**Why this priority**: 无工具的机器人只能依赖训练知识，无法回答"今天/最近"、"公司内部某文档"之类问题。这是从"玩具"到"生产力工具"的关键升级，但基础对话 (Story 1/2) 已可独立交付。

**Independent Test**: 向机器人提问一个明确需要联网的问题（如"最新发布的 iOS 版本号"），验证卡片内出现对应的工具调用条目且最终答案准确。

**Acceptance Scenarios**:

1. **Given** 机器人已启用网页搜索工具，**When** 用户问及时效性信息，**Then** 卡片内显示一条"网页搜索"调用（含查询关键词摘要）及其耗时与成功状态，正文基于搜索结果作答并标注来源链接。
2. **Given** 用户提供一个 URL 并要求总结，**When** 机器人识别到需要抓取该 URL，**Then** 卡片显示"网页抓取"工具调用，成功后给出基于页面内容的摘要。
3. **Given** 用户询问一篇飞书云文档的内容且已完成一次性 OAuth 授权，**When** 机器人代表该用户读取文档，**Then** 卡片显示"飞书文档读取"调用并返回文档正文摘要；若用户尚未授权，机器人回复一条引导链接，用户点击走完授权页后再次提问即可成功；若授权已过期/撤销，机器人引导用户重新授权。
4. **Given** 某次工具调用失败（网络、上游 API 错误等），**When** 失败发生，**Then** 卡片内该工具条目显示失败标识与简短错误摘要，但机器人仍尝试给出可用的回答或明确说明无法作答。

### User Story 5 - 通过 MCP 扩展外部工具生态 (Priority: P3)

运维/管理员配置 MCP (Model Context Protocol) 服务后，机器人自动接入这些服务提供的工具，用户无需感知底层差异即可使用。

**Why this priority**: 这是生态扩展点，让非开发者也能为机器人接入新能力。P1/P2 交付后再做可以覆盖绝大多数日常需求，故为 P3。

**Independent Test**: 在配置里启用一个 MCP 服务（例如文件系统或 GitHub MCP），重启机器人，用户通过自然语言触发其提供的工具，验证工具在 `/tools` 列表中可见、调用成功、结果正确回流。

**Acceptance Scenarios**:

1. **Given** 配置了一个有效的 MCP 服务，**When** 机器人启动，**Then** 该服务提供的工具出现在 `/tools` 列表中，工具名带有来源标识以避免与内置工具重名。
2. **Given** 某个 MCP 服务启动失败或断连，**When** 机器人启动或运行中失联，**Then** 机器人自身不崩溃；其他服务与内置工具仍可正常使用；`/tools` 列表中该服务标记为不可用。
3. **Given** MCP 工具调用超时或返回错误，**When** 错误发生，**Then** 按 Story 4 场景 4 的方式呈现失败，不影响其他工具调用。

### Edge Cases

- **同一用户并发消息**：用户在机器人上一条还未答完时又发一条 → 机器人应拒绝第二条并提示"上一条仍在处理中"，**不得**排队执行或混合输出。
- **超长回答**：机器人产出的正文超过飞书单卡片容量上限 → 必须分段推送，不得截断后静默丢失内容；所有段应在同一线程/楼中回复，用户能读到完整答案。
- **超长上下文**：累计会话过长导致即将超过模型上下文窗口 → 保证机器人仍能响应（可通过裁剪/压缩），不得返回"上下文过长"的原始报错给用户。
- **工具调用死循环**：模型反复请求同一个工具无法终止 → 达到预设步数上限后必须主动停止，向用户说明无法继续。
- **会话过期**：用户 24 小时未活跃后继续对话 → 历史已过期的情况下机器人从零开始新会话，而不是报错。
- **权限边界**：用户通过机器人索取其本人无权访问的飞书资源 → 机器人必须拒绝并给出明确原因，不得泄露任何内容。
- **非文本消息**：用户 @ 机器人时附带图片、文件、表情等 → 本迭代只处理文本部分，其他内容给出"暂不支持"的明确反馈。
- **机器人自身重启**：运行中重启 → 进行中的对话失败提示用户重试即可；已持久化的历史上下文在 24 小时内可继续使用。

## Requirements *(mandatory)*

### Functional Requirements

**消息入口与触发**

- **FR-001**: 系统 MUST 仅在 1-1 私聊、或在群聊中被 @ 的消息上触发响应；未被 @ 的群内消息必须被忽略。
- **FR-002**: 系统 MUST 在接收到一条需响应的消息后的 3 秒内对用户产生可见的回复（至少显示"处理中"占位），不得出现"静默"现象。
- **FR-003**: 系统 MUST 在解析消息时移除 @mention 自身的前缀文本，把纯净的用户意图传给 AI。

**对话上下文**

- **FR-010**: 系统 MUST 为每位用户独立维护对话上下文；群聊中不同用户之间的历史**不得**互相可见或互相影响。
- **FR-011**: 系统 MUST 将上下文保留 24 小时（滑动过期，最后一次活动起）。超过窗口的会话视为新会话。
- **FR-012**: 系统 MUST 在同一用户收到的多条并发消息之间串行化处理；同一用户上一条未完成时，新消息 MUST 被拒绝并给出明确提示。
- **FR-013**: 用户 MUST 能通过 `/clear` 指令立刻清空自身上下文。

**指令拦截**

- **FR-020**: 系统 MUST 在消息进入 AI 推理前拦截以下指令并直接处理：`/clear`、`/help`、`/model <name>`、`/tools`、`/whoami`。
- **FR-021**: 被拦截的指令 MUST NOT 被计入对话上下文，也 MUST NOT 触发 LLM 调用或工具调用。
- **FR-022**: `/model` 指令 MUST 允许用户切换至任何当前已启用的模型；对不存在的模型名 MUST 返回当前可用列表。
- **FR-023**: `/tools` 指令 MUST 列出所有当前可用工具及其来源（内置或外部），并标注任何暂时不可用的工具。

**流式呈现**

- **FR-030**: 系统 MUST 将 AI 的响应以单条消息卡片进行增量更新，而非分多条独立消息发送。
- **FR-031**: 卡片 MUST 能同时/分阶段展示三类内容：模型的"思考/推理"摘要、最终正文、工具调用的发生与结果。
- **FR-032**: 当正文开始产出时，系统 MUST 在视觉上淡化或折叠思考区，使用户把注意力集中到答案本身。
- **FR-033**: 卡片更新 MUST 遵守消息平台的速率限制，不得因刷新过快而被限流或产生丢字/乱序现象。
- **FR-034**: 若 AI 最终正文长度超过单卡片容量，系统 MUST 自动分段发送，分段之间保持可读顺序和上下文连续。

**工具调用**

- **FR-040**: 系统 MUST 在以下情境下向 AI 开放相应的工具能力：通用网页搜索、指定 URL 抓取、日期时间查询、飞书云文档的读取与搜索。
- **FR-041**: 每次工具调用的发生、入参关键字段摘要、完成状态（成功/失败）、耗时 MUST 对用户可见。
- **FR-042**: 工具调用的失败 MUST NOT 造成整轮对话终止；失败结果 MUST 被反馈回 AI，由 AI 决定下一步（重试、改用其他工具、或告知用户）。
- **FR-043**: 单次对话的"工具调用循环"步数 MUST 有硬上限；达到上限后系统 MUST 终止循环并向用户说明"无法继续"。
- **FR-044**: 任何涉及访问用户/组织资源（如飞书文档）的工具 MUST 仅返回该调用者本身有权访问的内容；无权资源 MUST 明确拒绝。
- **FR-045**: 访问飞书云文档类资源的工具 MUST 以**调用者本人的身份**（而非机器人应用身份）执行，即走 per-user OAuth 授权得到的用户身份凭证。
- **FR-046**: 当某用户首次触发需要用户身份凭证的工具调用时，系统 MUST 在不中断对话体验的前提下向该用户发送一次授权引导（例如私聊一条含授权链接的消息/卡片）；用户完成授权后 MUST 能继续原提问，不必从头重说。
- **FR-047**: 用户授权产生的凭证（access token / refresh token）MUST 被加密存储且仅供该用户本人的后续工具调用使用；任何跨用户共用 MUST 被禁止。系统 MUST 自动处理凭证过期（refresh），并在 refresh 失败时引导用户重新授权。
- **FR-048**: 用户 MUST 有方式撤销自己已授予机器人的权限（至少通过飞书平台侧的权限管理入口；机器人侧**可选**提供 `/revoke` 指令，本迭代不强制）。撤销后该用户的凭证 MUST 在下次使用时被识别为无效并触发重新授权流程。

**多模型支持**

- **FR-050**: 系统 MUST 支持同时配置多个 AI 模型并允许在运行时切换（见 FR-022）。
- **FR-051**: 切换模型 MUST 不丢失当前用户的对话历史；新模型从既有上下文继续。

**MCP 扩展**

- **FR-060**: 系统 MUST 允许管理员在配置中声明一个或多个 MCP 服务作为工具来源。
- **FR-061**: MCP 工具 MUST 与内置工具命名隔离，用户侧看到的工具名不冲突。
- **FR-062**: 任一 MCP 服务不可用（启动失败、断连）MUST NOT 导致机器人主体不可用；其他工具与服务仍需照常工作。

**可靠性与错误反馈**

- **FR-070**: 任何后端失败（LLM 超时、工具错误、飞书 API 错误）MUST 以对用户友好的语言在卡片内反馈，不得泄露栈信息或内部标识。
- **FR-071**: 系统 MUST 对每次对话生成一个可追溯的 trace 标识，用于运维排障；此标识对普通用户不可见，但在必要时可通过 `/whoami` 或管理员日志关联。
- **FR-072**: 系统在自身重启后 MUST 保证已持久化的用户上下文在 24 小时内可继续使用。

### Key Entities

- **Conversation Session**：一位用户在一个对话场景（私聊或某个群）的累积上下文；包含历史消息序列、当前选择的模型、最后活动时间；生存期 24 小时。
- **Message**：单条对话消息，区分来源角色（用户/助手/工具结果），助手消息额外携带思考内容与工具调用记录。
- **Tool**：一个可被 AI 调用的能力单元，包含名称、功能描述、入参契约、结果契约、来源（内置或来自某 MCP 服务）。
- **Tool Call**：某轮对话中一次具体的工具调用实例，包含工具名、入参、结果、耗时、成功/失败状态。
- **Model Profile**：一个可用的 AI 模型配置，包含别名、标识、能力标志（如是否支持"思考"展示）。
- **MCP Server Registration**：一条 MCP 服务接入配置，机器人运行时通过它加载外部工具；包含服务名、接入方式、连通状态。
- **User OAuth Credential**：某位飞书用户授予机器人以其身份访问飞书资源（文档等）所得到的凭证集合，包含 access token、refresh token、过期时间、授权 scope、所属用户标识；每位用户最多一份有效凭证，凭证仅供其本人调用。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从 @ 机器人发出问题到在卡片上看到首个可见内容（思考占位或首段文字）的时间 ≤ 3 秒（P95）。
- **SC-002**: 对于"不触发工具"的普通问答，用户从发问到看到完整正文结束的时间 ≤ 10 秒（P90）。
- **SC-003**: 同一群内两名用户并发对话时，两人各自的上下文和回答**无**任何交叉污染（抽样 100 次并发会话，污染率 = 0）。
- **SC-004**: 任何一次工具调用失败 **都** 能让 AI 继续响应（不至于让用户看到空卡片或系统级错误）——工具失败场景下的"用户收到可读回复"的比例 ≥ 99%。
- **SC-005**: 指令 `/clear`、`/model`、`/help`、`/tools`、`/whoami` 的响应时间 ≤ 1 秒（P99），且这些指令**从不**消耗 LLM 调用次数。
- **SC-006**: 同一用户连续消息"上一条未完成即发第二条"的场景中，100% 情况下第二条被明确拒绝，绝不出现回答混杂。
- **SC-007**: 机器人整体可用性 ≥ 99%（按月度计），单个 MCP 服务故障不影响机器人主体可用性。
- **SC-008**: 至少 80% 的首次使用者可在不阅读任何外部文档的情况下完成：@ 机器人、获取回答、清空上下文三项基础操作。
- **SC-009**: 需要用户身份凭证的工具首次触发时，从用户点击授权链接到授权成功并能继续提问，全流程 ≤ 60 秒（P90）；已授权用户的后续调用**无感知**（不需要再次授权，凭证 refresh 对用户不可见）。

## Assumptions

- 机器人运行在部署方已持有飞书机器人应用凭证（app_id / app_secret）的前提下；凭证申请流程不属于本 feature 范围。
- 消息平台（飞书）一直提供稳定的长连接事件订阅通道和消息卡片更新 API；若平台侧长时间降级，机器人自身无法替代。
- 会话上下文**不**要求永久保存或跨 24 小时后恢复；24 小时滑动过期是业务可接受的取舍。
- 本迭代默认至少有一个云端 LLM 模型可用（由部署方提供 API key）；多模型是可选增强而非硬性要求。
- 用户通过机器人索取飞书资源时，访问权限等价于"调用者本人"在飞书上的权限；机器人以 per-user OAuth 得到的 `user_access_token` 代表用户身份发起调用，**不会、也无法**越权。
- 首次调用需要用户身份凭证的工具前，用户**必须**完成一次 OAuth 授权；未授权的调用会被引导至授权流程而非直接失败。
- "Skill（技能系统）"明确不在本 feature 范围之内。
- 本 feature 不包含公开的 HTTP 管理后台、用户自助绑定页面、或账单/使用统计界面；这些若需要将是后续独立 feature。
- 工具调用的硬上限设定为单轮对话内的合理步数（具体数值在 plan 阶段确定），目的是防死循环，而不是限制合法的多步推理。

## Out of Scope (本迭代明确不做)

- Skill / 技能系统（本体、注册表、触发路由）。
- 沙箱代码执行工具（Python/JS 在容器内运行）。
- 面向运维/管理员的 Web 管理后台（用户侧 OAuth 引导链接本迭代交付，但不包含管理员 dashboard UI）。
- Prometheus 指标上报、管理 Web 界面（log 结构化已做，UI 暂不做）。
- 多语言界面（回复语言跟随 LLM 能力，不额外做本地化资源）。
- 机器人侧的 `/revoke` 指令（用户可通过飞书平台侧的授权管理入口撤销，已覆盖最低合规需求）。
